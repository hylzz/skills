# 用户故事示例与最佳实践（中文）

面向 B 端/平台类 PRD，功能需求以**字段表 + 业务规则**为主；面向终端用户的功能，可补充用户故事。

---

## 标准格式

```
作为 [用户角色]，
我希望 [执行动作]，
以便 [获得价值]。
```

**验收标准**（必须可测试）：

- [ ] 给定 [前置条件]，当 [操作]，则 [预期结果]
- [ ] 边界：[异常或空数据场景] → [系统行为]

---

## INVEST 原则（中文对照）

| 原则 | 含义 |
| --- | --- |
| 独立 | 可单独开发与验收 |
| 可协商 | 细节可与研发讨论调整 |
| 有价值 | 对用户或业务有明确收益 |
| 可估算 | 团队能给出工作量 |
| 足够小 | 一个迭代内可完成 |
| 可测试 | 有明确验收标准 |

---

## 按领域示例

### 平台能力类

```
作为 子系统管理员，
我希望 通过 API 将本系统的业务配置注册到统一平台，
以便 无需在多个业务系统中重复维护相同元数据。
```

验收标准：

- [ ] 首次注册时自动创建系统记录与默认配置
- [ ] 相同版本重复注册不覆盖已有资源
- [ ] 版本变更时增量更新配置项，并记录变更日志

### 数据同步 / 集成类

```
作为 平台运维人员，
我希望 定时从 [外部系统] 分页拉取 [业务数据] 并幂等写入 [目标系统]，
以便 下游业务能查询到一致、完整、可追溯的权威数据。
```

验收标准：

- [ ] 单条数据重复同步不产生重复记录
- [ ] 单条失败不影响同批次其他数据，且记录失败原因
- [ ] 同步日志可查询最近 N 次任务状态与条数

### 电商 / C 端

```
作为 网购用户，
我希望 按价格区间筛选商品，
以便 在预算内快速找到合适商品。
```

验收标准：

- [ ] 拖动价格滑块后列表在 1 秒内刷新
- [ ] 无结果时展示空状态与清除筛选入口

---

## 常见错误

| 避免 | 推荐 |
| --- | --- |
| 「系统要快、好用」 | 「列表首屏加载 P95 < 2s」 |
| 写实现细节（用 Redis 缓存） | 写行为结果（相同查询 5 分钟内返回缓存结果） |
| 一个故事包含登录+下单+支付 | 拆成独立故事 |
| 无验收标准 | 每条故事至少 2 条可测试条件 |

---

## 与 PRD 章节的对应关系

- **三、功能需求详述**：平台/B 端以表格与规则为主，可选附用户故事
- **七、第一版功能范围 → 验收标准**：第一期端到端交付 checklist
- 需求 ID 建议：`F-001`（功能）、`NFR-001`（非功能）、`O-001`（范围外）
