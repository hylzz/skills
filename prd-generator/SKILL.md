# PRD 生成器（简体中文）

## 概述

生成结构完整、可落地的**简体中文**产品需求文档，默认采用国内企业/研发协作常用的章节体系（项目概述 → 架构 → 功能详述 → 接口/数据模型 → 范围与验收）。

## 语言与输出规范

| 规则 | 说明 |
| --- | --- |
| **默认语言** | 全文简体中文（标题、正文、表格、验收标准） |
| **模板** | `references/prd_template.md` |
| **用户故事** | `references/user_story_examples_zh.md` |
| **指标框架** | `references/metrics_frameworks_zh.md` |
| **英文 PRD** | 仅当用户明确说「英文 PRD」「English PRD」时使用 `references/prd_template_en.md` |
| **项目内范例** | 若用户 @ 了仓库内已有 PRD（如 `prd/[示例PRD].md`），**优先对齐该文档的章节与表格风格** |
| **生成脚本定位** | `scripts/generate_prd.sh` 只用于快速生成中文 PRD 骨架；生成后必须继续结合对话信息补全背景、规则、指标、验收标准等“血肉” |

**禁止：** 在未要求英文时输出英文章节标题（如 Executive Summary、Problem Statement）或英文用户故事句式（As a … I want …）。

## 核心工作流

用户提出写 PRD 时，按以下步骤执行。

### 步骤 1：澄清与分析需求

先通过对话收集关键信息；用户已提供详细 brief 时可跳过非关键问题。

如果信息不足以写出可落地 PRD，**不要直接编造完整文档**。先输出：

- **需求理解**：用 3-5 条总结当前已知信息
- **关键假设**：明确哪些内容是暂定假设
- **待确认问题**：只问会影响范围、验收或方案的关键问题
- **建议一期范围**：给出初步本期包含 / 本期不包含建议

**必填信息：**

- 产品/功能名称
- 项目背景与要解决的问题
- 目标用户
- 业务目标
- 成功指标（如何衡量）
- 时间/约束（如有）
- 本期范围 vs 不在范围

**发现性问题（用中文提问）：**

```
1. 要解决的核心问题是什么？当前痛点有哪些？
2. 主要用户角色是谁？各自的核心诉求？
3. 关键业务目标是什么？与公司战略/OKR 如何对齐？
4. 有哪些技术或合规约束？
5. 怎样算成功？有哪些可量化指标？
6. 第一期交付的时间预期？
7. 明确不在本期范围的内容有哪些？
```

### 步骤 2：生成 PRD 骨架

用户确认方向后，可用以下两种方式生成骨架：

1. 直接基于 `references/prd_template.md` 撰写。
2. 用户希望快速落盘时，运行 `scripts/generate_prd.sh` 生成中文 PRD 骨架。

`scripts/generate_prd.sh` 的输出不是最终 PRD。脚本生成后必须继续完善：

- 将 `[占位符]`、`[规则 1]`、`待补充` 替换为具体业务内容
- 补充功能模块、字段表、业务规则、异常边界
- 补充接口/数据模型概要，或明确说明本期暂不涉及
- 把成功指标写成可量化口径
- 把验收标准写成 QA 可执行的检查项

### 步骤 3：生成 PRD 正文

基于 `references/prd_template.md` 生成文档，标准章节包括：

1. **一、项目概述** — 背景、目标、目标用户、成功指标（可选）
2. **二、系统总体架构** — 架构图、技术栈、核心流程、非功能需求概要
3. **三、功能需求详述** — 按模块拆分，含字段表、业务规则、异常边界
4. **四、接口设计（概要）**
5. **五、数据模型设计（概要）**
6. **六、不在此范围的功能**
7. **七、第一版功能范围** — 本期包含 / 本期不包含 / **验收标准**
8. **八、附录** — 术语、开放问题、风险、参考资料

文档开头须包含 **文档修改记录** 表格。

### 步骤 4：撰写用户故事与验收标准

- **B 端/平台/集成类**：以「三、功能需求详述」中的表格与规则为主；必要时在模块下附用户故事。
- **C 端/面向终端用户**：补充用户故事，格式见 `references/user_story_examples_zh.md`：

```
作为 [用户角色]，
我希望 [执行动作]，
以便 [获得价值]。

验收标准：
- [可测试条件]
```

第一期交付清单写在 **七、第一版功能范围 → 验收标准**。

### 步骤 5：定义成功指标

参考 `references/metrics_frameworks_zh.md`，指标须：

- 用中文描述
- 可量化（百分比、延迟、成功率、数量等）
- 区分本期 vs 后续阶段（如有分阶段）

### 步骤 6：校验与评审

可选运行校验脚本（支持中文章节检测）：

```bash
scripts/validate_prd.sh <prd_file.md>
```

自检清单：

- [ ] 全文为简体中文（除专有名词、API 路径、技术栈英文名）
- [ ] 含「一、项目概述」「三、功能需求」「七、第一版功能范围」
- [ ] 范围边界清晰（六、不在此范围 + 七、本期不包含）
- [ ] 验收标准可测试
- [ ] 无未填写的 `[占位符]` 或大量 TBD

## 使用模式

### 模式 1：新功能 PRD

**用户：**「给 [某功能] 写一份 PRD」

1. 用中文做发现性提问  
2. 输出需求理解、关键假设和待确认问题  
3. 按模板或脚本生成 PRD 骨架  
4. 补全「三、功能需求」中的核心功能、业务规则、异常边界等内容  
5. 在「七、验收标准」列出可验证条目  

### 模式 2：平台 / 集成类 PRD

**用户：**「写从 [外部系统] 同步 [业务数据] 到 [目标系统] 的 PRD」

1. 明确数据流向、权威数据源、分阶段目标  
2. 先对齐本期范围、外部系统边界和验收口径  
3. 强化「二、架构」「四、接口」「五、数据模型」  
4. 用表格描述字段、同步策略、幂等与失败处理  
5. 参考用户提供的范例 PRD 骨架（若有）  

### 模式 3：精简 PRD

**用户：**「小需求，一页纸 PRD」

保留：**一、项目概述** + **三、功能需求（单模块）** + **七、范围与验收标准**，其余章节可省略或合并。

### 模式 4：英文 PRD

**用户：**「Create an English PRD for …」

使用 `references/prd_template_en.md` 与 `references/user_story_examples.md`，全文英文。

## 写作规范

### 优质需求

- **具体**：避免「快速」「好用」「灵活」等模糊词  
- **可测**：QA 能根据验收标准判定完成  
- **有价值**：每条需求能对应用户或业务收益  
- **有边界**：六、七章节写清不在范围的内容  

### 避免

- 未经验证的假设当作已定事实  
- 在 PRD 中过度指定实现（除非技术方案已评审锁定）  
- 第一期范围混入「规划」功能却不标注阶段  

### 图表

- 架构、数据流：优先 `plaintext` ASCII 或 mermaid  
- 复杂状态机：mermaid flowchart / sequenceDiagram

## 示例

**用户：**「按 [示例 PRD] 的结构，写 [某业务功能] 的初版 PRD」

**执行：**

1. 读取用户指定的范例 PRD，对齐章节编号与表格风格  
2. 用中文撰写；涉及外部系统、竞品或第三方平台时，若只是举例须注明「以 XX 为例」  
3. 先输出需求理解、待确认问题和建议一期范围  
4. 可用 `scripts/generate_prd.sh` 生成骨架，再继续补全正文  
5. 分阶段目标放入「一、项目概述」  
6. 同步、幂等、日志等写入「三、功能需求详述」  
7. 第一期范围与验收标准写入「七」  
8. 输出到用户指定路径（如 `prd/xxx.md`）

## 故障排查

| 问题 | 处理 |
| --- | --- |
| 文档过长 | 用精简模式，二期内容标「规划」单独列表 |
| 需求含糊 | 先输出需求理解、关键假设、待确认问题，再补充数字、示例 JSON、状态表 |
| 脚本生成后仍像模板 | 继续替换占位符，补全业务规则、字段、异常边界、成功指标和验收标准 |
| 混用中英文标题 | 统一改为中文章节名 |
| 校验脚本报缺章节 | 确认使用中文标题（如「一、项目概述」） |
